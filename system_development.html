01 — Foundations

What is a System?

A system is a set of organised items/components that interact in a given environment and within specific boundaries in order to achieve a desired goal or objective. A system is described as being either soft or hard.

Soft Systems

Human activity systems — their boundaries may be fluid or keep changing, their goals and objectives often conflict, and it is difficult to precisely define exact measures of performance.

Example: Political system

Hard Systems

Systems whose goals and objectives are clearly defined and whose outcomes are predictable and can be modelled accurately.

Example: Stock management system

Open vs Closed Systems: An open system receives input from and gives output to the environment. A closed system does not interact with the environment at all.
02 — Characteristics

Characteristics of Systems

Holistic Thinking Sub-systems Process Objectives / Purpose Environment Boundaries Interdependence Feedback Control Inputs & Outputs System Entropy
CharacteristicExplanation
Holistic Thinking A system is considered as a whole. The combination of various components creates a complex whole whose overall goals are more sophisticated than those of individual components.
Sub-systems A system is made up of different components (sub-systems). A system does not exist in isolation — it may itself be a component of a larger system.
Process The element of a system that does the actual transformation of input into output, changing the system from one state to another.
Objectives / Purpose Objectives may be real or stated. An organisation must state its objectives and operate to achieve them. e.g. a school system's objective is to enable students excel academically.
Environment The surrounding within which an organisation operates, providing a reason for the system's existence.
Boundaries The external elements whose changes in attitude, behaviour and property affect the system's state — it defines the limit of the system.
Interdependence For a system to be effective and efficient, its components or sub-systems must be co-ordinated and linked together.
Feedback & Control Control is the method by which a system adapts to environmental changes to produce expected output. Feedback is a check to see whether predetermined goals are being met and to detect deviation.
Inputs & Outputs A system communicates with its environment by receiving inputs and giving outputs.
System Entropy Entropy means decay. Systems naturally decay over time — becoming useless due to improvements in technology, new management directives, or changing user requirements. They must be reviewed and improved periodically.
03 — Classification

System Classification

TypeDescriptionExample
Deterministic Systems whose outputs are known precisely by their inputs. Computer system
Probabilistic Systems whose outputs can only be predicted but not known precisely. Business and economic systems

04 — Information Systems

Information Systems

An information system is a set of persons, procedures, technologies and other resources that collects, transforms and disseminates information in an organisation. It can also be defined as an arrangement of people, data, processes and information that work together to support and improve day-to-day operations and decision making.

Purposes of an Information System

  • To provide people within an organisation with information to assist in decision making.
  • To provide responsive service to meet user needs.
  • To enable sharing of information.
  • To facilitate interaction and dissemination of data throughout the various levels of an organisation.
  • To support information processing by enhancing data collection, processing and communication.
05 — Need for New Systems

Why Develop a New Information System?

Three main circumstances necessitate the development of a new information system:

Directives

New requirements imposed by government or management that introduce changes the organisation must implement to meet its needs, e.g. improving quality of service.

Problems in the Current System

Undesirable circumstances in the existing system that prevent the organisation from meeting its goals, e.g. slow or poor service delivery.

New Opportunities

When an opportunity arises that could enable the organisation to improve its services, e.g. the availability of new technology.

06 — System Analyst

Role of the System Analyst

A system analyst is a person responsible for identifying an organisation's needs and problems, then designing and developing an information system to solve them.

  • Reviewing the existing system and making recommendations on how to improve or implement an alternative.
  • Working hand in hand with programmers to construct a computerised system.
  • Co-ordinating training for new system users and owners.
07 — Methodologies

System Development Methodologies

System development methodologies refer to a combination of theories and techniques used to develop information systems. These theories form the basis for the techniques used to undertake the System Development Life Cycle (SDLC).

Traditional Approach

Relies mostly on the skills and experience of individual staff members carrying out the project. The manual system is replaced with a computerised one without changing the overall structure of the former system. As a result, weaknesses of the former system are not addressed — they are forwarded into the new system.

Rapid Application Development (RAD)

Evolved from the theory that businesses today heavily rely on IT, so development must be quick enough to maintain a competitive advantage. RAD relies on prototyping — a design model presented to the user for selection. Other approaches include SWAT (Small team With Advanced Tools) and JAD (Joint Application Development).

Main Problem with RAD: The system may have oversights and weaknesses due to the speed of development.

Structured Approach

Defines a set of clearly documented stages to be followed when developing a system. Each stage specifies the activities to be carried out by the system analyst and their team. This is the most thorough and widely recommended approach.

08 — SDLC

Stages of System Development (SDLC)

  1. Problem Recognition & Definition
    The analyst decides if the project is necessary and thoroughly defines the problem, identifying any constraints. Ends with a feasibility study.
  2. Information Gathering
    The analyst gathers data about the current system and user needs using record inspection, interviews, questionnaires, observation, or automated methods.
  3. Requirements Specification
    Detailed requirements for the new system are defined: output, input, files/data stores, and hardware/software requirements.
  4. System Design
    The physical design of the system is detailed. Tools used include flowcharts, data flow diagrams, entity-relationship models, structured charts, pseudo-codes, decision tables and trees.
  5. System Construction
    Coding, installation and testing of modules. Done by programmers using 3GL languages (Pascal, COBOL), 4GL languages (Visual Basic, Delphi), or customised ready-made packages.
  6. System Implementation
    Installing the system, testing it, converting from the old system, and training users. Involves file creation/conversion, changeover, and staff training.
  7. System Review & Maintenance
    Adjustment, enhancement or correction of errors after implementation. Review checks whether the system still meets original specifications. If not, development may start again.
09 — Feasibility Study

Feasibility Study

A special study carried out at the end of problem definition to establish the costs and benefits of the proposed project. Feasibility is assessed in four ways:

Operational Feasibility

Establishes the extent to which users are comfortable and happy with the proposed or new system.

Technical Feasibility

Establishes whether available technology is sufficient or can be upgraded, and whether staff have the relevant technical skills.

Economic Feasibility

Establishes whether the new system is cost-effective by analysing all costs and benefits of the proposed system.

Schedule Feasibility

Establishes whether development of the proposed system can be accomplished within the available time frame.

10 — Information Gathering

Methods of Information Gathering

Record Inspection

Documents maintained in connection to the system are referenced. Forms a good starting point for the analyst to gain an overall insight of the existing system.

✓ Advantages
  • Provides basic background information about the system
  • Helps design interview questions and questionnaires
  • Helps decide who to interview
✗ Disadvantages
  • Documents may be obsolete and not reflect current status
  • Human systems change — documents may give a false picture
  • Documents are often bulky and time-consuming to read
  • Documentation may be poor or incomplete

Interview

A face-to-face communication between the analyst (as interviewer) and affected personnel (as interviewee), with questions geared to getting the information needed.

✓ Advantages
  • Analyst has access to first-class information
  • Makes interviewee feel part of the system
  • Gives adequate opportunity for opinions to be put forward
  • Non-verbal communication (facial expressions) can be observed
  • Questions can be rephrased instantly for clarification
✗ Disadvantages
  • Difficult to organise and time-consuming
  • Interviewee may not fully open up on personal or sensitive issues
  • Many analysts are poor interviewers, risking failure
  • May not be practical due to location of interviewees
  • Interviewee may give opinions rather than proven facts
  • Interviewee may be suspicious of the proposed system's intentions
Proxemics: Issues related to physical contact and sitting arrangements during an interview. Proximity between interviewer and interviewee can affect comfort and openness.

Questionnaires

A document containing standard questions distributed to a large number of people to gather information from those affected by the proposed system.

When to Use Questionnaires

  • When limited information is required from a large number of people.
  • When there is a large number of people involved in the system project.
  • When the people are geographically dispersed.
✓ Advantages
  • Answers filled in a less tense environment
  • Privacy leads to more sincere responses
  • Provides many people the opportunity to give input
  • Saves time — especially where respondents are geographically scattered
  • Respondents fill at their own convenience
✗ Disadvantages
  • A slow way of gathering facts
  • Difficult to prepare — requires expertise
  • Questions can be easily misunderstood, leading to ambiguous replies
  • Analysis of results requires a lot of time
  • Respondents may not fill or return questionnaires

Observation

The observer participates or watches closely as a person performs activities in order to learn about the system. Gives the analyst first-hand experience of problems and exposes them to system requirements.

✓ Advantages
  • Concepts too difficult for non-technical staff to explain can simply be observed
  • Helps the analyst become acquainted with operational personnel
  • Provides "know how" needed to assist in recommended changes
  • Can correct hazy or erroneous impressions of the system
✗ Disadvantages
  • Costly and time-consuming — requires the observer to be on-site
  • Staff are uncomfortable when being watched
  • The person observed might alter behaviour, leading to wrong requirements

Automated Methods

Used when actual data is required but difficult to obtain through interviews, observation or questionnaires. Data is collected using devices that automatically capture data from the source.

Examples include scanners, video cameras, and tape recorders. Particularly useful in areas that are not easily accessible.

11 — Requirements Specification

Requirements Specification

The system analyst must produce detailed requirements for the new system across four main areas:

Output Specification

Concerns the generation of reports to facilitate decision making (in hardcopy or softcopy). Factors to consider:

  • Target audience — users may need transaction reports; management needs overall performance summaries.
  • Frequency — some reports are required daily, others weekly, monthly or annually.
  • Quality and format — information must be clear and well-structured.
  • Cost — output must be produced at a reasonable cost.
  • Mode of output — e.g. softcopy via screen or hardcopy via printer.

Input Specification

Factors to consider:

  • The contents and volume of input.
  • The mode and devices of input selected.
  • Layout and sequence of input.
  • Nature of the system — batch processing uses periodical input; real-time systems use on-demand input.
  • User interface mock-ups or dummy forms can be used to identify system inputs.

File / Data Stores (Database)

The analyst must identify the number of files needed, determine the structure of each file, and identify the attributes of records. An attribute is a unique characteristic of a record for which a data value can be stored — attributes become fields in database tables.

Factors to consider when designing files:

  • Record key field — a unique attribute for each record.
  • Data type — e.g. Text for book titles, Date/Time for borrowing dates.
  • Length of each field — e.g. 30 characters for names, 10 for numbers.
  • Backup and recovery strategies — updated copies must be stored separately to enable reconstruction if the current file is corrupted.

Hardware & Software Requirements

The analyst must specify all hardware and software requirements, considering:

  • Economic factors — e.g. price.
  • Operational factors — e.g. reliability, upgradability, and compatibility with existing resources.
  • User-friendliness.
12 — Design & Construction

System Design & Construction

System Design

Involves detailing the physical design of the system — the how of the process. Once the logical design is outlined during system analysis, the analyst determines the physical design describing the hardware, software and operating procedures required. Tools used include:

Flowcharts Data Flow Diagrams Entity Relationship Models Structured Charts Pseudo-codes Decision Tables Decision Trees

System Construction

Refers to coding, installation and testing of modules and their components (outputs, inputs and files). Done by programmers. Programming techniques include:

  • 3GL high-level structured languages — e.g. Pascal, COBOL.
  • 4th generation languages — e.g. Visual Basic, Visual COBOL, Delphi Pascal.
  • Customised ready-made standard packages — e.g. database software, financial packages, enterprise management systems.
Note on System vs Program Flowcharts: A system flowchart covers the entire system while a program flowchart covers a single module or part. Several program flowcharts may be derived from one system flowchart.
13 — Implementation

System Implementation & Changeover

Testing the System

Test data is entered to verify outputs are as expected. The system is tested against both the requirements specification and the design specification to confirm all requirements are met.

File Creation & Conversion

Setting up master files to support the new system — either from scratch or by converting files from the old system (e.g. manual files to electronic files). File conversion refers to changing files from one format to another.

Staff Training

Training aims to convince staff of the system's effectiveness, remove fear of change, and enable staff to cope with the new processing tasks. Methods of training include:

Demonstration Film Show Manual Reference Group Discussion Visits

Changeover Strategies

Direct / Straight Changeover

The new system commences live operation and the old system is immediately abandoned.
✓ Advantage

Cheap — no need to run two systems in parallel.

✗ Disadvantages
  • Extremely risky
  • Places the organisation in a do-or-die situation

Parallel Changeover

Both the old and new systems run alongside each other simultaneously.
✓ Advantages
  • Provides a method of training on the new system
  • Personnel become familiar prior to actual changeover
✗ Disadvantages
  • Costly to run two systems
  • Cross-checking is difficult
  • Requires more staff and training

Phased Changeover

Implementation is done one part/phase at a time. A trial phase (piloting) may run in one department before full roll-out.
✓ Advantages
  • Allows quick implementation of phases
  • Less costly
  • Failures are limited in scope
  • Causes minimal disruption
✗ Disadvantage
  • Danger of incompatibility between hardware or software elements of the same system
14 — Maintenance & Documentation

System Review, Maintenance & Documentation

Review & Maintenance

System maintenance is the adjustment, enhancement, or correction of errors after implementation. System review involves going through the specifications and testing the system to find out whether it still meets the original requirements. If not, system development may start over again.

System Documentation

  1. Report of fact finding — outlines methods used to collect data, weaknesses of the current system, and recommendations.
  2. Requirements specification — details the output, input, hardware and software required for the new system.
  3. System flowchart — shows the overall functionality of the proposed information system.
  4. Tables / file structures — file and table structure definitions used in system construction.
  5. Sample test data — used to test whether the computerised information system works as expected.
  6. Output reports — sample outputs from various modules proving the system is delivering desired results.
  7. User manual — helps the user operate the system with minimal guidance, including: how to install and start the system, the interface, how to carry out various tasks, error correction and how to get help.
15 — Revision

Review & Revision Questions with Answers

Define the term "information system".
An information system is the arrangement of people, data, processes and information that work together to support and improve the day-to-day business operations of an organisation.
Differentiate between soft systems and hard systems.
In soft systems, goals and objectives are not clearly defined, boundaries are fluid, and performance is hard to measure precisely. In hard systems, goals and objectives are clearly defined and outcomes are predictable and can be modelled accurately.
What is system control? Why do we need feedback in a system?
System control is the method by which a system adapts to changes in the environment in order to give the expected output or perform at the expected level. Feedback is needed in order to adjust controls for better performance of the system — it checks whether predetermined goals are being met and detects deviation.
Explain the term "system boundary".
System boundary is the scope or limit within which a system's components operate — the external element whose changes in attitude, behaviour and property affect the state of the system.
Explain three system development theories/methods.
  • Traditional approach — relies on individual staff skills and experience; replaces the manual system with a computerised one without changing the overall structure, so weaknesses are carried forward.
  • Rapid Application Development (RAD) — relies on prototyping to develop and implement systems quickly. Main disadvantage is that the system may have weaknesses due to speed of development.
  • Structured approach — defines a set of clearly documented stages each with specific activities. The most thorough approach to system development.
Define the term "feasibility study" as used in system development.
A feasibility study is a special study carried out to establish the costs and benefits of a new system. It assesses operational, technical, economic and schedule feasibility.
State two advantages of interview over questionnaires.
  • Non-verbal communication (such as facial expressions) can be used and observed.
  • Questions can be rephrased instantly for clarification and to probe the interviewee further.
Outline the four areas considered during the requirements specification stage.
  • Output specification
  • Input specification
  • Files / data stores
  • Hardware and software requirements
State four factors to consider in order to design a good file.
  • Record key field — a unique attribute for each record.
  • Data type for each field.
  • Length of each field.
  • Backup and recovery strategies — to enable reconstruction if the file is corrupted.
Explain the three tasks carried out during system implementation.
  • File creation and conversion — setting up master files from scratch or converting existing files (e.g. manual to electronic).
  • Staff training — training users to convince them of the system's effectiveness, remove fear of change, and enable them to cope with new tasks.
  • Changeover — transitioning from the old system to the new one using direct, parallel, or phased strategies.
Give an advantage of straight changeover over parallel changeover.
Straight changeover is cheaper to implement because you do not have to run two systems simultaneously.
Define the term "attribute" as used in system development.
An attribute refers to a unique characteristic of a particular entity or record for which a data value can be stored in the system database. Attributes become fields in database tables.
Outline the contents of a user manual that would help the user run the system with minimal guidance.
  • How to install and start the system.
  • The interface of the system.
  • How to carry out various tasks.
  • Test data and expected output.
  • Error correction and troubleshooting guide.
